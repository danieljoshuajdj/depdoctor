Solution Steps:

Implement satisfiesPeerRequirement Helper: Introduce a utility function, e.g.:

ts
Copy
/**
 * Determine if installedVersion satisfies the peerRange.
 * Strips workspace/file/link prefixes and handles semver ranges properly.
 */
function satisfiesPeerRequirement(installedVersion: string, peerRange: string): boolean {
  // Remove any workspace/link/file prefix (globally).
  const cleanRange = peerRange.replace(/(?:workspace:|link:|file:)/g, '');
  // If range is empty or wildcard, treat it as satisfied.
  if (!cleanRange || cleanRange === '*') return true;
  // Coerce installed version (handles things like "19.x").
  const coerced = semver.coerce(installedVersion);
  if (!coerced) return false;
  // Validate range syntax first.
  if (!semver.validRange(cleanRange)) {
    // If range is something like a git commit, warn or skip.
    return false;
  }
  // Use semver.satisfies with prerelease included.
  return semver.satisfies(coerced.version, cleanRange, { includePrerelease: true });
}
This uses semver.satisfies(version, range) from the official SemVer library. Ensure all prefix patterns (workspace:, file:, link:) are removed globally (replace(/.../g)), not just at string start. Use semver.validRange() to check for range validity.

Replace Existing Logic: Wherever the code currently marks a peer mismatch, use satisfiesPeerRequirement. For example, in analyzer.upgradeRisk, change the peer conflict check to:

ts
Copy

  // Report conflict
}
Handle Empty/Missing Peers: If a peer dependency entry exists but the installed version is missing entirely, treat this separately. For required peers, that is a conflict; for peerOptional (see below), it should be a warning but not a hard conflict. Use [npm’s peerDependenciesMeta] to detect optional peers:

json
Copy
"peerDependenciesMeta": { "pkg": { "optional": true } }
If a peer is marked optional and not installed, report it as optional rather than mismatch. (See Testing below.)

2. Aging Engine (Dependency Age Calculation)
Current Issue: The aging logic reported impossible values (e.g. thousands of years old) because it relied on incomplete metadata and naive math. We must ensure all age calculations are finite and plausible.

Solution Steps:

Audit Files: Check src/health/aging.ts and src/scanner/package-intelligence.ts. Identify where publishDate, ageDays, and averageAge are computed. Search for any NaN or negative values entering calculations.

Fetch Real Publish Dates: Use pacote.packument(pkgName@version, { fullMetadata: true }) to retrieve the packument with full metadata. According to the package registry spec, when fullMetadata: true the response includes a "time" field mapping versions to publish times. For example:

ts
Copy
const pack = await pacote.packument(`${name}@${version}`, { fullMetadata: true });
const publishTime = pack.time[version]; // ISO string
If publishTime exists, parse it into a Date. Use new Date(publishTime) or Date.parse() which supports ISO8601.

Safe Date Handling: Check that the parsed date is valid:

Use Number.isFinite() to verify numeric results (MDN notes this is the reliable way). E.g. if (!Number.isFinite(pubDate.getTime())) skip;.
If pubDate > now (future date) or pubDate < 0 (before epoch), treat as invalid.
Only compute ageDays = (Date.now() - pubDate) / (1000*60*60*24) if pubDate <= now.
If ageDays is NaN or > 36500 (100 years, say), discard it. Cap or fallback.
Fallback Hierarchy: If registry data is unavailable (package not on registry, offline, or missing field), apply fallbacks:

Known Ages Map: For popular packages, use a curated map (e.g. React, lodash, typescript, etc.) with approximate first-release dates. (For instance, React first released ~2013, TypeScript ~2012). Add tests to verify these values.
Version Heuristic: If no mapping, infer age from major version: e.g. assume each major bump ~2 years old, or use a formula like ageYears = (major - 1) * 2. This is crude but avoids zero.
Minimum Age: Do not report zero age. If all else fails, use a conservative minimum (e.g. 90 days).
The key is never report insane values or NaN. If metadata is missing or corrupted, skip the package in averages entirely.

Average Age Calculation: When aggregating:

ts
Copy
const validAges = allPackages.map(p => p.ageDays).filter(age => Number.isFinite(age));
const totalDays = validAges.reduce((a,b) => a+b, 0);
const averageAgeDays = validAges.length ? totalDays/validAges.length : 0;
Use Number.isFinite again to avoid infinity. For reporting, convert averageAgeDays to years+days.

workerd	1665248 years (bug)	e.g. 1.6 years
react-dom (v18.2.0)	0.0 years (3 days)	~4 years
Average	7874.8 years	~2.7 years

The “impossible” values must vanish. (Tables here illustrate the fix; actual values will come from code.)
